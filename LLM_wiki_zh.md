# LLM Wiki

一种使用大语言模型构建个人知识库的模式。

这是一个想法文件，设计目的就是让你可以把它复制粘贴给自己的 LLM Agent，例如 OpenAI Codex、Claude Code、OpenCode / Pi 等。它的目标是传达高层思路，具体实现则由你的 agent 和你协作完成。

## 核心想法

多数人与 LLM 和文档互动的体验都像 RAG：你上传一组文件，LLM 在回答问题时检索相关片段，然后生成答案。这当然有用，但 LLM 每次提问都在从零重新发现知识，没有积累。只要你问一个需要综合五份文档的细微问题，LLM 就必须每次都重新找到并拼接相关片段。没有任何东西被沉淀下来。NotebookLM、ChatGPT 文件上传，以及大多数 RAG 系统都是这种方式。

这里的想法不同。LLM 不只是查询时从原始文档中检索，而是**增量构建并维护一个持久 wiki**：一组结构化、相互链接的 Markdown 文件，位于你和原始来源之间。当你加入一个新来源时，LLM 不只是把它索引起来留待以后检索。它会阅读它，抽取关键信息，并把它整合进已有 wiki：更新实体页面，修订主题摘要，记录新数据在哪里与旧说法冲突，强化或挑战正在演化的综合判断。知识只需要编译一次，然后保持更新，而不是每次查询都重新推导。

关键差异在于：**wiki 是一个持久、会复利增长的产物。**交叉引用已经在那里。矛盾已经被标出。综合判断已经反映了你读过的一切。你每加入一个来源、每提出一个问题，wiki 都会变得更丰富。

你几乎从不需要亲自写 wiki，LLM 会负责写作和维护。你负责选择来源、探索方向，以及提出正确的问题。LLM 做所有繁琐工作：摘要、交叉引用、归档、记账，而这些正是让知识库随着时间推移真正有用的部分。实践中，我会一边打开 LLM agent，另一边打开 Obsidian。LLM 根据我们的对话做编辑，我实时浏览结果：跟随链接、查看图谱、阅读更新后的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这种模式可以应用在很多不同语境中。举几个例子：

- **个人**：追踪自己的目标、健康、心理、自我提升；归档日记、文章、播客笔记，并随着时间构建一个关于自己的结构化图景。
- **研究**：在数周或数月内深入一个主题；阅读论文、文章、报告，并增量构建一个带有演化论点的综合 wiki。
- **读一本书**：一边读一边归档每一章，为人物、主题、情节线以及它们之间的关系建立页面。读完时你会得到一个丰富的伴读 wiki。可以想象像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki：由志愿者多年共同维护，包含人物、地点、事件、语言等成千上万的互链页面。你也可以在个人阅读时构建类似的东西，让 LLM 负责所有交叉引用和维护。
- **商业或团队**：由 LLM 维护的内部 wiki，输入来自 Slack 线程、会议记录、项目文档、客户通话。也可以让人类参与审核更新。因为 LLM 做了团队里没人想做的维护工作，wiki 才能保持最新。
- **竞品分析、尽职调查、旅行规划、课程笔记、爱好深挖**：任何你会长期积累知识，并希望它们被组织起来而不是四处分散的场景。

## 架构

这里有三层：

**原始来源**：你精心收集的一组来源文档，包括文章、论文、图片、数据文件等。这一层不可变，LLM 只读取它们，绝不修改它们。这是你的事实来源。

**wiki**：一个由 LLM 生成的 Markdown 文件目录。里面有摘要、实体页、概念页、比较页、总览页、综合页。LLM 完全拥有这一层。它创建页面，在新来源到来时更新页面，维护交叉引用，并保持整体一致。你阅读它；LLM 编写它。

**schema**：一个文档，例如 Claude Code 使用的 `CLAUDE.md` 或 Codex 使用的 `AGENTS.md`，用来告诉 LLM 这个 wiki 如何组织、有哪些约定，以及在摄入来源、回答问题或维护 wiki 时该遵循哪些工作流。这是关键配置文件：它让 LLM 成为一个有纪律的 wiki 维护者，而不是一个通用聊天机器人。随着你们逐渐弄清楚什么适合当前领域，你和 LLM 会共同演化这个 schema。

## 操作

**摄入。**你把一个新来源放进原始集合，并告诉 LLM 处理它。一个示例流程是：LLM 阅读来源，和你讨论关键要点，在 wiki 中写一个摘要页，更新索引，更新整个 wiki 中相关的实体页和概念页，并向日志追加一条记录。单个来源可能触及 10 到 15 个 wiki 页面。就我个人而言，我更喜欢一次摄入一个来源并保持参与：我阅读摘要，检查更新，并指导 LLM 强调哪些内容。但你也可以用更少监督批量摄入很多来源。具体工作流应由你发展出适合自己风格的版本，并写入 schema，供未来会话沿用。

**查询。**你针对 wiki 提问。LLM 搜索相关页面，阅读它们，并带引用地综合出答案。答案可以根据问题采取不同形式：Markdown 页面、比较表、幻灯片（Marp）、图表（matplotlib）、canvas。重要洞见是：**好的答案可以作为新页面归档回 wiki。**你要求的一次比较、一段分析、一个新发现的联系，本身都很有价值，不应该消失在聊天记录中。这样，你的探索会像摄入的来源一样，在知识库中复利增长。

**检查。**定期让 LLM 给 wiki 做健康检查。可以查看：页面之间的矛盾、已被新来源取代的陈旧说法、没有入链的孤立页面、被多次提到但还没有独立页面的重要概念、缺失的交叉引用、可以通过网页搜索补齐的数据空白。LLM 很擅长建议新的研究问题和新的来源方向。这能让 wiki 在增长时保持健康。

## 索引和日志

有两个特殊文件可以帮助 LLM 和你在 wiki 增长后继续导航。它们用途不同：

**index.md** 面向内容。它是 wiki 中所有内容的目录：每个页面列出链接、一句话摘要，并可选地包含日期或来源数量等元数据。它按类别组织，例如实体、概念、来源等。LLM 会在每次摄入时更新它。回答查询时，LLM 先读索引以找到相关页面，再深入阅读这些页面。在中等规模下，例如约 100 个来源、数百个页面，这种方式出奇有效，也避免了必须搭建基于 embedding 的 RAG 基础设施。

**log.md** 面向时间。它是一份只追加的记录，说明发生了什么、何时发生：摄入、查询、检查等。一个有用的小技巧是：如果每条记录都以一致前缀开头，例如 `## [2026-04-02] ingest | Article Title`，日志就可以用简单的 Unix 工具解析；`grep "^## \\[" log.md | tail -5` 可以给出最近 5 条记录。日志为 wiki 的演化提供时间线，也帮助 LLM 理解最近做过什么。

## 可选：CLI 工具

到某个阶段，你可能会想构建一些小工具，帮助 LLM 更高效地操作 wiki。最明显的是 wiki 页面搜索引擎：小规模时索引文件就够了，但随着 wiki 增长，你会想要真正的搜索。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是面向 Markdown 文件的本地搜索引擎，支持混合 BM25 / 向量搜索和 LLM 重排序，而且全部在设备本地运行。它既有 CLI，LLM 可以通过 shell 调用，也有 MCP server，LLM 可以把它当作原生工具使用。你也可以自己构建更简单的版本；需要时 LLM 可以帮你临场写一个朴素搜索脚本。

## 技巧

- **Obsidian Web Clipper** 是一个浏览器扩展，可以把网页文章转换成 Markdown。它非常适合快速把来源放入原始集合。
- **把图片下载到本地。**在 Obsidian 的 Settings -> Files and links 中，把 “Attachment folder path” 设为一个固定目录，例如 `raw/assets/`。然后在 Settings -> Hotkeys 中搜索 “Download”，找到 “Download attachments for current file”，并绑定一个快捷键，例如 Ctrl+Shift+D。剪藏文章后按下快捷键，所有图片都会下载到本地磁盘。这是可选的，但很有用，因为它允许 LLM 直接查看和引用图片，而不是依赖可能失效的 URL。注意，LLM 不能原生地一次读完带内联图片的 Markdown；变通方式是让 LLM 先读文本，再单独查看部分或全部被引用图片来获得更多上下文。这有点笨拙，但足够好用。
- **Obsidian 的图谱视图** 是观察 wiki 形状的最佳方式：哪些内容连接到哪些内容，哪些页面是枢纽，哪些页面是孤岛。
- **Marp** 是一种基于 Markdown 的幻灯片格式。Obsidian 有对应插件。它适合直接从 wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可以基于页面 frontmatter 运行查询。如果你的 LLM 会给 wiki 页面添加 YAML frontmatter，例如标签、日期、来源数量，Dataview 就能生成动态表格和列表。
- wiki 本质上只是一个 Markdown 文件组成的 git 仓库。你天然获得版本历史、分支和协作能力。

## 为什么这行得通

维护知识库最烦人的部分不是阅读或思考，而是记账。更新交叉引用，保持摘要不过时，记录新数据何时与旧说法冲突，在几十个页面之间维持一致性。人类会放弃 wiki，因为维护负担增长得比价值更快。LLM 不会厌倦，不会忘记更新交叉引用，并且能在一次处理中触及 15 个文件。wiki 能保持维护，是因为维护成本接近于零。

人类的工作是筛选来源、引导分析、提出好问题，并思考这一切意味着什么。LLM 的工作是其他所有事情。

这个想法在精神上与 Vannevar Bush 1945 年提出的 Memex 有关：一种个人化、精选的知识存储，其中有文档之间的关联路径。Bush 的愿景比后来的 Web 更接近这里描述的模式：私有、主动整理，并且文档之间的连接与文档本身同样有价值。他当时无法解决的问题是：谁来维护。LLM 解决了这一点。

## 说明

本文刻意保持抽象。它描述的是一种想法，而不是某个具体实现。确切的目录结构、schema 约定、页面格式和工具链都取决于你的领域、偏好以及你选择的 LLM。上面提到的一切都是可选和模块化的：挑有用的，忽略无用的。比如，你的来源也许只有文本，所以完全不需要图片处理；你的 wiki 也许足够小，一个索引文件就够了，不需要搜索引擎；你也许不关心幻灯片，只想要 Markdown 页面；你也可能想要一组完全不同的输出格式。正确用法是把这份文档交给你的 LLM agent，并与它一起实例化一个适合你需求的版本。这份文档唯一的任务是传达这个模式。剩下的，你的 LLM 可以自己推演出来。
