1. 总体判断：不要一上来做“大而全 Agent”
我建议把设计原则定成：

先做一个能闭环的任务型 Agent，再逐步加记忆、多工具、多 Agent、自我改进。

这和 Anthropic 的实践建议一致：很多成功的 agentic system 不是靠复杂框架，而是靠简单、可组合的模式；只有当任务需要灵活决策、不能预先写死流程时，才值得上真正的 Agent，否则 workflow 更稳定。

一个清晰的 Agent 最小闭环应该是：

用户目标
  ↓
任务理解 / 范围确认
  ↓
计划生成
  ↓
工具调用 / 信息收集 / 环境交互
  ↓
状态更新
  ↓
缺口判断 / 是否继续
  ↓
产出结果 / 报告 / 文件 / 操作
  ↓
评测与日志
对应到实现上，不要先堆很多角色，而是先实现一个干净的 runtime loop：

while not state.done and state.step < max_steps:
    action = planner.decide_next_action(state)
    observation = tool_executor.run(action)
    state = state_updater.update(state, action, observation)
    state.done = gate.should_stop(state)

return final_writer.render(state)
核心不是“多 Agent”，而是 state 是否清楚、tool 是否好用、stop/gate 是否可靠、结果是否可评估。

技术深度可以放在三点

第一，检索不是一次性搜索，而是基于 open questions 迭代搜索。第二，不是简单 summarization，而是抽取 method/dataset/metric/limitation/代码可用性。第三，报告必须带引用和 evidence trail，每个关键结论能回到原文或网页。

评估方法

找 5 个你熟悉的研究主题，让 Agent 生成调研报告。评估：

Coverage：是否覆盖核心论文
Precision：推荐论文是否相关
Citation accuracy：引用是否支持结论
Gap usefulness：gap 是否真的有启发
Time saved：相比人工调研节省多少时间
Reuse rate：你是否愿意继续用它跟踪领域
这个方案可以借鉴 ReAct、Reflexion 和 Voyager：ReAct 给你“边推理边行动”的基本工具调用循环；Reflexion 给你“失败后写反思并进入记忆”的机制；Voyager 的 automatic curriculum + skill library + iterative prompting 很适合迁移成“研究技能库持续积累”。

先说结论
Claude Code 的“梦境模式”更准确地说是 AutoDream / Dream memory consolidation：它不是让 Claude 半夜自动写代码，而是让 Claude 在空闲或手动触发时，对长期记忆做一次“睡眠式整理”——把旧的、重复的、矛盾的、相对时间失效的记忆清理掉，并把真正有用的信息写回更干净的 memory 文件。

但要注意：截至我这次检索，Anthropic 官方文档明确公开的是 Auto Memory、/memory、Skills、Subagent memory、Hooks、Auto mode/Routines 等机制；“AutoDream”本身更像是近期社区围绕 Claude Code 新版本、UI 暴露项、GitHub issue、博客分析和开源 /dream skill 形成的说法，并没有一个稳定的官方完整说明页。官方文档说 Auto Memory 会把 Claude 学到的 build commands、debugging insights、architecture notes、code style preferences 等保存到本地 memory；MEMORY.md 前 200 行或 25KB 会在每次会话开始时加载，topic files 则按需读取。

1. “梦境模式”到底解决什么问题？
它解决的不是“Claude 记不住”，而是：

Claude 记了太多低质量东西，导致长期记忆开始污染后续行为。

Claude Code 的 Auto Memory 会在多次会话中积累项目知识和用户偏好，但长期使用后会出现几类典型问题：

1. 旧决策没有被删除
   例如之前说用 MySQL，后来改成 PostgreSQL，但两个记忆都还在。

2. 相对时间失效
   例如“下周五前 freeze merge”，几周后就变成无意义信息。

3. 重复记忆膨胀
   同一个偏好被不同措辞保存了多次。

4. 记忆索引过大
   MEMORY.md 变长，占用启动上下文，反而挤压当前任务空间。

5. debug 临时经验污染长期记忆
   一次临时 workaround 被当成长期规则。
Claude Code 官方 memory 文档已经暗示了这个结构：MEMORY.md 是 memory directory 的索引，应该保持简洁；详细内容应放到 topic files 中，Claude 会按需读取。
 Dream/AutoDream 本质上就是对这个 memory layer 做 offline maintenance。

2. 它的核心流程：Orient → Gather → Consolidate → Prune
社区版 /dream skill 把流程写得很清楚：它安装为 ~/.claude/skills/dream/SKILL.md，运行后做 4 个阶段：Orient、Gather、Consolidate、Prune。该项目 README 也明确说 /dream 是一个 memory consolidation skill，会审查最近 session，把信息提炼成 durable memories。

Phase 1：Orient，先理解当前记忆结构
读取：

~/.claude/projects/<project>/memory/
├── MEMORY.md
├── debugging.md
├── api-conventions.md
└── ...
目标不是直接写新记忆，而是先搞清楚：

当前有哪些记忆？
MEMORY.md 是不是已经太长？
哪些 topic files 存在重复？
哪些记忆看起来过时？
社区 /dream skill 的源码说明会先创建/读取 memory directory，读取 MEMORY.md，再 skim existing topic files，避免制造重复记忆。

Phase 2：Gather，从近期会话里找“值得沉淀的信号”
它不是把所有 transcript 都 summarize 一遍，而是找 non-obvious durable signal：

用户明确纠正过什么？
用户确认过什么方案是对的？
最近有没有新的项目决策？
有没有外部系统指针，例如 dashboard、issue tracker、Slack channel？
有没有旧 memory 被当前代码或近期会话推翻？
社区版 /dream 会扫描最近 session transcripts 的 .jsonl，但强调不要无脑读完整 transcript，而是 narrow grep、只找已经怀疑重要的信息。
 这个设计很重要：Dream 不是“总结一切”，而是“提取可复用的长期约束和偏好”。

Phase 3：Consolidate，合并、更新、替换矛盾记忆
这一步是核心。

好的 memory consolidation 不应该只是 append，而应该：

新事实覆盖旧事实
重复偏好合并成一条
相对时间转成绝对时间
临时信息不进入长期记忆
有来源、有理由、有适用条件
社区 /dream skill 要求每条 memory 使用 YAML frontmatter，包含 name / description / type，并区分 user / feedback / project / reference 四类；它还明确要求将 “yesterday / last week” 这类时间转为绝对日期，并删除被推翻的事实。

一个合理的记忆格式大概长这样：

---
name: feedback_testing_strategy
description: User prefers minimal focused regression tests before broad refactors
type: feedback
---

Rule: Before large refactors, add one focused regression test around the failing behavior.

Why: The user wants changes to be grounded in observable behavior, not broad speculative cleanup.

How to apply: When asked to refactor or debug, first identify the narrow failing path and add/inspect a regression test.
Phase 4：Prune + Index，把 MEMORY.md 控制成“索引”而不是“垃圾桶”
MEMORY.md 应该像目录，不应该像全文数据库。社区 /dream skill 明确要求把 MEMORY.md 控制在 200 行以内、约 25KB 以内，每条索引一行，详细内容放到 topic file 里。
 这和 Claude Code 官方 memory 机制吻合：官方文档说启动时只加载 MEMORY.md 的前 200 行或 25KB，topic files 不在启动时加载，而是按需读取。

也就是说，Dream 的核心不是“更多记忆”，而是：

更小的入口索引 + 更干净的 topic files + 更少的矛盾事实。

3. 官方 AutoDream、社区 /dream、Auto Memory 是什么关系？
我建议这样理解：

名称	状态	本质
Auto Memory	官方明确文档化	Claude 自动保存跨会话经验
/memory	官方明确文档化	查看、编辑、开关 memory
AutoDream	社区和部分 UI/issue 中出现	自动/后台 memory consolidation
/dream skill	社区可安装实现	手动触发的 Dream 工作流
Auto Mode	官方文档化	权限审批自动化，不是 memory consolidation
Routines	官方文档化	云端定时/事件触发自动任务，不是 Dream 本身
GitHub 上有人在 Anthropic 的 claude-code issue 里报告：/memory 菜单显示 AutoDream 开启，~/.claude/settings.json 中有 autoDreamEnabled: true，但 /dream 返回 “Unknown skill: dream”，没有真正触发后台 dreaming task。这个 issue 说明至少在某些版本/账号/平台上，UI、设置、命令执行路径之间并不稳定。

所以我的判断是：

AutoDream 是一个正在浮出水面的 Claude Code memory maintenance 方向，但最好不要依赖隐藏环境变量或泄漏代码；更稳妥的方式是用官方 Skills/Hooks/Memory primitives 自己复刻一个可审计的 /dream。

Claude Code 官方 Skills 文档支持把一个 SKILL.md 变成 slash command，例如 /skill-name；skill body 只有在使用时加载，适合把重复 workflow 从 CLAUDE.md 中迁出去。
 这正好是 /dream 的合理实现方式。

4. 为什么这个设计有研究价值？
我觉得 Dream 模式真正有价值的地方，不是“像人睡觉”这个比喻，而是它把 Agent memory 从一个被动存储层，变成了一个 有生命周期的系统对象。

传统做法是：

每次对话 → 追加一点 memory → 下次全部喂给模型
Dream 的做法是：

每次对话 → 产生原始经验
一段时间后 → 离线整理
整理结果 → 只保留可迁移、可验证、可复用的信息
下次会话 → 用更小、更准的 memory 入口启动
这和你关注的 Agent / Harness Evolution 很接近：不是更新模型参数，而是更新 agent harness 的外部认知结构。Memory 不是日志，Memory 是 runtime 的一部分。

更进一步说，Dream 模式体现了 4 个系统设计思想：

1. Memory is not storage
   记忆不是越多越好，而是要有选择、压缩、验证、淘汰。

2. Reflection should be offline
   不要在主任务 loop 里塞太多反思；让后台/手动 maintenance loop 做整理。

3. Index and evidence should be separated
   启动时只加载索引，需要时再读取详细 evidence。

4. Consolidation must resolve contradiction
   不能 append-only，否则旧错事实会长期污染 agent。
这一点也和近期关于 context files 的研究相互呼应。ETH Zurich 等人的 AGENTS.md 研究发现，repository-level context files 往往会降低任务成功率，并使推理成本增加超过 20%；他们的结论是，人工写的 context files 也应该只保留 minimal requirements。
 另一篇 2026 年 4 月的规则研究则发现，负约束如“不要重构无关代码”更有用，而正向指导如“遵循代码风格”可能反而伤害表现。

这说明 Dream 的方向是对的：不要把所有经验都塞进上下文，而是把长期有用、不可从代码直接推断、且有明确适用条件的信息留下来。

5.2 Memory schema
我会比社区版再多加几个字段，尤其是 provenance 和 validity：

---
id: mem_feedback_testing_strategy_20260425
type: feedback
scope: project
description: User prefers focused regression tests before broad refactors
created_at: 2026-04-25
updated_at: 2026-04-25
source:
  - session: 2026-04-25T10-30-00.jsonl
    turn: 18
confidence: high
validity: active
verify_before_use: false
---

Rule: Add or inspect focused regression tests before large refactors.

Why: The user cares about behavior-grounded changes and dislikes speculative broad cleanup.

How to apply: For debugging/refactor tasks, first identify the narrow behavior that must be preserved.
几个字段很关键：

source：未来可以追溯，不要让 memory 变成无来源断言
confidence：避免把一次性猜测当成事实
validity：active / stale / superseded / archived
verify_before_use：涉及文件、API、命令、外部系统时必须先验证
5.3 触发条件
可以先做手动触发：

/dream
然后再做自动触发：

def should_dream(state):
    return (
        state.sessions_since_last_dream >= 5
        or state.memory_index_lines > 150
        or state.contradiction_score > 0.3
        or state.days_since_last_dream >= 3
    )
不要一开始就“全自动后台跑”。先手动稳定，再加 idle trigger。

5.4 Consolidation prompt 要点
Dream prompt 不要写成“总结最近会话”，而要写成“维护 memory 数据库”。

核心指令应该是：

你不是在总结对话。
你是在维护长期记忆。

只保存未来会话中仍有价值、不可从当前代码/文档直接推断的信息。

保存：
- 用户长期偏好
- 用户纠正过的工作方式
- 已确认的项目决策
- 外部系统指针
- 反复出现的失败模式

不要保存：
- 当前任务临时状态
- 代码结构和文件路径，除非它们是外部不可推断事实
- git history
- debug fix recipe
- 已经写在 CLAUDE.md 的规则
- 没有来源的推测

遇到冲突：
- 优先当前代码/当前配置
- 其次近期明确用户指令
- 再其次旧 memory
- 不确定则标记为 needs_review，不要强行合并
5.5 写入流程要做成可审计
不要直接让 LLM 随便改 memory。建议流程：

1. LLM 生成 proposed changes
2. Validator 检查 YAML/frontmatter/路径/长度
3. 生成 diff
4. 可选人工确认
5. 原子写入
6. 自动备份旧 memory
7. 输出 summary
伪代码：

def run_dream(project_root: Path):
    memory = load_memory(project_root)
    sessions = load_recent_sessions(project_root, since=memory.last_dream_at)

    candidates = extract_memory_candidates(sessions, memory)
    verified = verify_candidates(candidates, project_root, memory)
    patch = build_memory_patch(memory, verified)

    validate_patch(patch)
    backup_memory(memory)
    apply_patch_atomically(patch)

    return DreamReport(
        created=patch.created,
        updated=patch.updated,
        merged=patch.merged,
        pruned=patch.pruned,
        needs_review=patch.needs_review,
    )
6. 几个相关方案
A. 社区 /dream skill
sathwick-p/dream 是最直接的社区实现：安装后提供 /dream slash command；README 说只安装一个 ~/.claude/skills/dream/SKILL.md 文件，另有可选 enhanced memory extraction，把更积极的记忆抽取规则追加到 CLAUDE.md。
 它的优点是简单、易懂、可复制；缺点是依赖 prompt 约束，缺少强验证和 provenance 机制。

B. lossless-code
lossless-code 是另一个方向：它不主张删除，而是用 SQLite + DAG 保存所有消息、层级摘要和 source lineage；它提供 lcc_grep / lcc_expand / lcc_context / lcc_dream 等工具，强调“summary 可以追溯回原始消息”。
 这个方向更适合研究型系统，因为它解决了 Dream 的一个潜在问题：prune 之后怎么证明没有误删？

C. ContextCov / executable guardrails
ContextCov 不是 Dream，但它对 memory/rules 系统很有启发：它把自然语言 instruction files 转成可执行 guardrails，包括 AST 检查、shell shim、架构验证器。
 对 Dream 来说，关键启发是：重要规则不应该永远只是 markdown；高风险规则应该逐渐“毕业”为可执行检查。

D. Claude Code Hooks / Subagent Memory
Claude Code 官方 subagent memory 支持让 subagent 拥有持久 memory directory，并在系统提示中加入读写 memory 的说明；官方 hooks 则允许在 SessionStart、Stop、PreToolUse、PostToolUse、PreCompact、PostCompact 等生命周期事件触发逻辑。
 如果你自己实现 Dream Worker，hooks 是比隐藏环境变量更干净的接入点。

7. 风险：Dream 也可能制造“记忆污染”
这点很重要。Dream 如果做不好，会把短期错误固化为长期偏见。

近期一篇关于 Claw/heartbeat background execution 的安全论文指出，后台执行中遇到的不可信内容可能进入短期上下文，随后被写入长期记忆，并影响后续用户交互；论文把它形式化为 Exposure → Memory → Behavior 路径，并报告了长期记忆污染和跨会话影响。

所以 Dream Worker 必须有几条安全原则：

1. 所有 memory 必须有 provenance
2. 外部网页/issue/slack 信息默认低信任
3. 用户明确确认 > agent 自己总结
4. 涉及当前代码状态的 memory 使用前必须重新验证
5. pruning/merging 必须保留 diff 和 backup
6. 不要让后台 Dream 隐形运行而无日志
社区粘贴的 AutoDream 内部代码片段声称其使用 forked subagent、querySource: auto_dream、skipTranscript: true 等机制；这类设计如果属实，对产品体验是“干净”的，但对审计不一定好。
 如果你自己做系统，我反而建议 不要 skip transcript，至少保留 dream report 和 memory diff。

Dream Worker = offline retrospective memory maintainer

Claude Code 的梦境模式真正有意思的地方，是它把 Agent 的长期记忆从 append-only 笔记，变成了一个会定期自我维护、自我去噪、自我纠错的外部认知层。


如果我想做一个个人的信息源agent，需要如何设计，请参考别人的设计思路，同时也要有创新性，以满足作业的要求
此外，这个知识库需要支持个人定制化，也就是能提供给其他同学使用（研究方向可能有不同）
对于我个人而言，我是人工智能专业的博士研究生，研究方向是大模型和智能体，同时也关注国内外人工智能相关的进展（不局限于大模型和智能体），需要每天定时爬取推送

我会把它设计成一个“个人科研情报 Agent”，而不是普通 RSS 聚合器：核心是信息源采集、个性化画像、去重与证据核验、知识库沉淀、每日推送、反馈学习六个闭环。接下来我会结合现有产品/框架的思路，给你一版能做作业、也能真实自用的方案。

我已经把可借鉴对象分成两类：一类是 Feedly/Folo/Readwise 这种“信息入口产品”，另一类是 ResearchRabbit/Semantic Scholar/Hugging Face Papers 这种“科研发现系统”。你的作业最好不要复刻其中任何一个，而是做一个“可配置研究画像 + 每日情报闭环 + 可追溯知识库”的组合体。

Thought for 54s
下面这版我建议你作为作业项目来设计，名字可以叫：

ResearchRadar：面向个人研究方向的每日信息源 Agent

它不是普通 RSS 阅读器，也不是论文摘要工具，而是一个 可定制的个人科研情报系统：每天自动从论文、GitHub、技术博客、AI 新闻、会议平台等来源抓取信息，根据用户研究画像过滤、去重、排序、总结，并把真正值得看的内容推送给用户，同时沉淀到个人知识库中。

1. 先参考别人的设计，但不要照抄
可以借鉴的现有系统大概有几类。

Feedly AI / AI Feeds 的思路是：不是把所有来源都推给用户，而是用 AI 模型从大量来源中筛出符合用户关注主题的内容。Feedly 文档明确说 AI Feed 不等于传统 RSS Feed，它会从开放网络中筛选匹配特定主题、威胁、实体或趋势的信息。这个很适合借鉴为“个性化信息过滤层”。

Folo + RSSHub 的思路是：用开放 RSS 生态统一接入多种来源。RSSHub 是开源 RSS 生成器，覆盖大量网站和平台；Folo 则定位为 AI RSS Reader，强调从互联网噪声中筛出用户真正关心的知识。这个适合借鉴为“信息源接入层”。

Readwise Reader / Ghostreader 的思路是：把 RSS、网页、PDF、邮件 newsletters 等阅读材料统一到一个阅读工作流里，并提供 AI 阅读助手。Readwise Reader 官方页面提到它支持 RSS、PDF、网页高亮等，Ghostreader 是它的 AI assistant。这个适合借鉴为“阅读 + 标注 + 知识沉淀层”。

ResearchRabbit / Semantic Scholar Recommendations 的思路是：不是只按关键词搜论文，而是从已知论文、引用关系、相似论文、后续工作中扩展研究地图。ResearchRabbit 强调通过已有论文探索相关文献；Semantic Scholar Recommendations API 支持根据输入论文返回推荐论文。这个适合借鉴为“科研发现层”。

Hugging Face Daily Papers / Trending Papers 的思路是：把社区关注度作为信号，把每日论文和趋势论文推送给用户。Hugging Face Papers 页面本身提供 Daily / Weekly / Monthly trending papers，并支持邮件订阅。这个适合借鉴为“热度与社区反馈信号”。

但你的创新点应该是：把这些能力整合成一个可配置的个人信息源 Agent，并且让每个同学可以拥有不同研究画像。

2. 核心定位：不是“信息聚合器”，而是“个人研究情报员”
我建议你的作业目标写成：

本项目实现一个面向研究者的个人信息源 Agent。系统每天定时从论文平台、代码平台、AI 新闻源、技术博客和社区热点中采集新信息，根据用户研究画像进行个性化过滤、去重、排序、摘要和推送，并将高价值内容沉淀到可检索的个人知识库中。系统支持不同用户配置不同研究方向，从而适配多位同学的个性化科研信息需求。

这个定位比“每天爬取 AI 新闻并总结”更有作业价值，因为它包含了：

1. 信息源采集
2. 个性化画像
3. 语义过滤
4. 去重与聚类
5. 证据化摘要
6. 每日推送
7. 知识库沉淀
8. 用户反馈学习
3. 总体架构
可以设计成 7 层：

用户配置层
  ↓
信息源采集层
  ↓
清洗与标准化层
  ↓
个性化过滤与排序层
  ↓
Agent 分析层
  ↓
知识库沉淀层
  ↓
每日推送与反馈层
更具体一点：

ResearchRadar
├── Source Connectors        # arXiv / OpenReview / Semantic Scholar / GitHub / RSS / blogs / news
├── Normalizer               # 统一格式、清洗正文、提取元信息
├── Deduplicator             # URL / title / embedding / arXiv id 去重
├── Personal Relevance Ranker# 根据用户画像排序
├── Research Agent           # 摘要、分类、判断重要性、生成行动建议
├── Knowledge Base           # 向量库 + 结构化数据库 + 原文缓存
├── Daily Digest Generator   # 生成每日推送
└── Feedback Learner         # 根据点赞、收藏、忽略、点击更新用户画像
Agent 设计上不建议一开始做复杂多 Agent。Anthropic 的 agent 设计建议是：很多成功系统不是靠复杂框架，而是靠简单、可组合的模式；只有任务确实需要动态决策时才引入更强的 agentic loop。

所以你的 MVP 可以是 workflow-first, agent-enhanced：

先用稳定 workflow 完成每日采集、过滤、摘要、推送；
再在关键节点引入 Agent 判断，例如“这篇为什么重要”“和我的研究有什么关系”“是否值得深读”。

6. 核心创新点设计
我建议至少做 4 个创新点，作业展示时会更有辨识度。

创新点 1：Research Intent Profile，不只是关键词订阅
普通 RSS 是：

关键词 = agent
只要文章里有 agent 就推送
你的系统应该是：

研究画像 = 主题 + 方法偏好 + 排除项 + 近期任务 + 已读论文 + 反馈历史
也就是说，它不仅知道你关注 “agent”，还要知道你更关心：

agent architecture
agent progress estimation
agent memory
future prediction agent
self-evolving agent
research agent
tool-use agent
而不那么关心：

real estate agent
marketing agent
customer support agent
这一步可以用 embedding + LLM 分类器实现。

创新点 2：Novelty-aware Ranking，只推“对你新的东西”
普通系统会推：

今天最火的 10 篇论文
你的系统应该推：

今天对你最有信息增量的 10 条内容
排序公式可以这样设计：

FinalScore =
  0.35 * RelevanceToProfile
+ 0.20 * NoveltyToUserMemory
+ 0.15 * SourceAuthority
+ 0.15 * TrendSignal
+ 0.10 * Actionability
+ 0.05 * Diversity
其中：

RelevanceToProfile：和用户研究画像是否相关
NoveltyToUserMemory：和用户已读内容是否重复
SourceAuthority：来源是否可靠
TrendSignal：是否在社区、GitHub、Hugging Face、OpenReview 中升温
Actionability：是否有代码、数据、benchmark、可复现实验
Diversity：避免 10 条全是同一个方向
这比“热度排序”更适合作业，因为它体现了个性化智能。

创新点 3：Evidence Card，每条推送都可追溯
不要只生成一句：

这篇论文很重要。
要生成结构化卡片：

### 1. Paper Title

**一句话结论**：
这篇论文提出了一个新的 Agent memory compression 方法，用于降低长任务中的上下文成本。

**为什么和你相关**：
它直接关联你关注的 agent memory、long-horizon agent、context management。

**关键贡献**：
- 提出 memory pruning 策略
- 在 long-horizon benchmark 上降低 token 成本
- 提供开源实现

**证据**：
- arXiv abstract
- GitHub repo
- Hugging Face discussion
- OpenReview comment

**建议动作**：
- 今日快速读 abstract + method
- 如果做 agent harness，可以重点看 memory update 机制
这样它就不是“AI 摘要”，而是“有证据的研究情报”。

创新点 4：Feedback-to-Profile Learning
每天推送后，让用户点：

👍 有用
👎 不相关
📌 收藏
📖 深读
🧊 暂时忽略
系统将反馈更新到用户画像：

如果用户连续收藏 agent memory 相关内容：
  提高 agent memory 权重

如果用户连续忽略纯 benchmark 榜单：
  降低 pure benchmark 权重

如果用户收藏某些作者：
  加入 author watchlist
这一步是“个人信息源 Agent”和普通日报脚本的分界线。

7. Agent 运行流程
每日定时任务可以这样跑：

每天 08:00
  ↓
读取所有用户 profile
  ↓
为每个用户生成当天 source query
  ↓
抓取新内容
  ↓
标准化成 InformationItem
  ↓
去重
  ↓
粗筛：关键词 + embedding
  ↓
精筛：LLM 判断 relevance / novelty / actionability
  ↓
生成每日 digest
  ↓
推送到 Telegram / Slack / Email / Web
  ↓
用户反馈
  ↓
更新用户画像和知识库
更像 Agent 的版本：

DailyResearchAgent
  1. Decide what to look for today
  2. Collect candidate items
  3. Cluster and deduplicate
  4. Judge relevance for each user
  5. Write personalized digest
  6. Update memory from feedback
OpenAI Agents SDK 把 agent 定义为能够规划、调用工具、协作并保持足够状态完成多步工作的应用；LangGraph 则强调 durable execution、长期/短期 memory 和 human-in-the-loop，适合这种每天运行、可恢复、有状态的系统。

def score_item(item: InformationItem, profile: UserProfile) -> dict:
    relevance = compute_embedding_similarity(item, profile.primary_topics)
    negative = compute_embedding_similarity(item, profile.negative_topics)
    novelty = compute_novelty(item, user_history=profile.user_id)
    authority = compute_source_authority(item.source)
    trend = compute_trend_signal(item)
    actionability = compute_actionability(item)

    final_score = (
        0.35 * relevance
        - 0.25 * negative
        + 0.20 * novelty
        + 0.15 * authority
        + 0.15 * trend
        + 0.10 * actionability
    )

    return {
        "final_score": final_score,
        "relevance": relevance,
        "novelty": novelty,
        "authority": authority,
        "trend": trend,
        "actionability": actionability,
    }

14. 作业展示时可以强调的“Agent 感”
为了满足作业里对 Agent 设计的要求，你不能只展示一个爬虫 + 摘要器。你需要强调它有以下 agentic 能力：

14.1 自主决定今天查什么
不是每天固定关键词，而是根据：

用户画像
近期收藏
近期反馈
热点变化
未覆盖方向
动态生成 query。

例如今天发现你连续收藏了 “agent progress estimation”，系统明天会自动加大：

agent progress estimation

14.2 自主解释为什么推荐
每条内容不只给摘要，还要给：

为什么和你有关
和你已有知识有什么差异
建议你怎么处理

用户反馈后，系统更新：

topic weight
source preference
author watchlist
negative topics

每周整理一次用户收藏、忽略和深读记录；

把短期热点降权；

1. 要不要爬 X.com？
结论：要，但不要一开始“广泛爬 X”
X 对 AI 研究进展非常有价值，因为很多信息会先出现在 X 上：

1. 新模型发布
2. 论文作者解读
3. 开源项目更新
4. benchmark 争议
5. 公司研究员的非正式观点
6. 产品发布和 API 变更
7. 社区对论文/工具的第一轮反馈
但 X 有三个问题：

1. 噪声极高
2. API 成本和限制较多
3. 不适合无边界抓取和长期全文存储
X 官方文档现在明确提供 X API，可访问 posts、users、spaces、lists、trends 等，并采用 pay-per-use 计费；它也强调应使用官方 API、遵守 rate limits，而不是 scraping/browser automation。
 X Developer Agreement 还限制不得绕过 rate limits、不得把 X API 或 X Content 用于训练/微调 foundation/frontier model，并对内容再分发、地理位置数据、token 安全等有约束。

所以设计上不要叫“爬 X”，更准确地叫：

X Watchlist Connector：基于官方 API 的账号/列表/关键词监听器。

第一版可以只抓这几类：

1. 你手动维护的研究者账号列表
2. 官方实验室账号
3. 重要开源项目账号
4. 关键词搜索：LLM agent、code agent、reasoning、RL for LLM、OpenAI、Anthropic、DeepSeek、Qwen 等
5. X Lists：比如 AI Researchers / LLM Infra / Agents
不要做：

1. 全站爬取
2. 绕登录抓取
3. 浏览器自动化爬 X
4. 大规模存储原始 tweet 全文
5. 用 X 内容训练或微调模型
更稳妥的存储策略是：

存：
- post id
- author
- url
- timestamp
- metrics
- short excerpt
- system-generated note
- related paper/repo/url

不长期存：
- 大量原文全文
- 用户画像级隐私数据
- 地理位置数据
- 可替代 X 的内容镜像
X 的作用应该是 early signal，不是最终事实源。也就是说，如果 X 上有人说“某个模型 SOTA 了”，系统应该继续查论文、GitHub、blog、leaderboard，而不是直接推给你。

2. 要不要接 Hacker News？
结论：非常应该，建议 v1 就做
Hacker News 比 X 更适合作为第一版信息源，因为它：

1. 有官方 API
2. 接入成本低
3. 适合技术趋势、开源工具、AI infra、agent 工程
4. 评论区经常有高质量技术讨论
5. 对软件工程和产品化方向特别有帮助
Hacker News 官方 API 通过 Firebase 暴露近实时公开数据，URI 前缀是 https://hacker-news.firebaseio.com/v0/；官方文档说 v0 API 本质上是公开数据结构 dump，item 包括 story、comment、job、poll 等，并且可以读取 title、url、score、time、kids、descendants 等字段。
 另外，Algolia 也提供 Hacker News Search API，适合按关键词检索历史帖子和评论。

HN 在你的系统里应该承担三种角色：

1. Trend Signal
   看 AI/infra/code agent 相关内容是否进入技术社区讨论。

2. Discussion Signal
   抓取高质量评论，识别开发者关注点、质疑点、真实使用反馈。

3. Cross-check Signal
   当一篇论文/项目在 arXiv、GitHub、X、HN 同时出现，说明它更值得关注。
例如：

论文源：arXiv 上出现一篇 Agent Memory 论文
GitHub：对应 repo 一天内涨星
X：作者发 thread 解读
HN：有人讨论它是否真的 useful

=> 系统判定为“高优先级深读”
3. 信息源优先级建议
我建议这样排：

v1 必做：
- arXiv
- GitHub
- Hugging Face Papers
- RSS / 官方博客
- Hacker News

v1.5 选做：
- X.com official API
- OpenReview
- Semantic Scholar
- Papers with Code
- alphaXiv

v2 再做：
- Reddit
- YouTube / podcast transcript
- 微信公众号 / 中文社区
- Slack / Discord / 邮件 newsletter
原因是：v1 需要稳定、可复现、低成本。HN、arXiv、GitHub、RSS 更适合先跑通系统；X 很有价值，但更适合作为加分项，因为它涉及 API 成本、政策限制和噪声控制。

4. 大模型交互层：必须做
你提到的这个功能非常关键：

用户看到某条推送后，可以直接问大模型：“这篇论文讲了什么？”“和我的研究有什么关系？”“有没有类似工作？”“我能不能基于它做一个 project？”
然后系统把有价值的交互沉淀进个人知识库。

这会让系统从“每日信息推送”变成：

Personal Research Copilot over Your Information Stream

也就是：

每日信息流
  ↓
个性化推荐
  ↓
用户主动追问
  ↓
LLM 深入解释
  ↓
交互内容沉淀
  ↓
更新个人知识库和兴趣画像
  ↓
第二天推荐更准
这是非常好的作业创新点。

5. 交互层应该怎么设计？
我建议做三种对话入口。

5.1 针对单条信息问
用户点开一篇论文、一条 GitHub repo、一条 HN 讨论、一条 X post，然后问：

这篇论文的核心贡献是什么？
它和 Voyager / Reflexion / MemGPT 有什么关系？
这篇 work 对我的 future prediction agent 有帮助吗？
如果我要复现，最小实验怎么做？
它的 claim 可靠吗？
系统回答时不能只靠大模型常识，而要基于当前 item 的内容、链接、摘要、评论、相关论文和用户画像。

流程：

用户问题
  ↓
读取当前 item
  ↓
检索相关知识库内容
  ↓
必要时读取原文 / repo / comments
  ↓
LLM 生成带证据回答
  ↓
保存问答摘要
5.2 针对每日 digest 问
用户可以问：

今天最值得我读的是哪三条？
今天有没有和 agent progress estimation 有关的内容？
这些动态里哪些可能形成 research idea？
今天有哪些只是 hype，可以跳过？
这对应的是 Daily Digest Q&A。

5.3 针对整个个人知识库问
用户可以问：

最近两周 agent memory 有什么新进展？
有哪些论文和 Claude Code memory / dream mode 有关？
帮我整理一个 activation-level intervention for tool-use agents 的阅读路线。
我最近收藏的内容里，有没有能形成项目选题的方向？
这对应的是 Personal Research KB Q&A。

这三层对话入口可以统一成一个接口，但上下文范围不同：

Item Chat      = 当前条目 + 相关资料
Digest Chat    = 今日 digest + 当日候选池
Research Chat  = 全部知识库 + 用户画像 + 历史交互
6. 关键：不要把所有聊天记录都塞进知识库
交互沉淀不能简单保存完整对话，否则很快污染知识库。

应该分三层保存：

Raw Conversation Log
  保存完整聊天，用于追溯，不默认进入推荐画像

Distilled Research Note
  从对话中抽取稳定、有用的信息

Profile Update Candidate
  判断这次对话是否说明用户兴趣发生变化
例如你问：

“这篇论文和我的 future prediction harness 有什么关系？”
大模型回答后，系统不应该把整段聊天都当知识。它应该沉淀成：

## Note: Agent memory consolidation may be relevant to future prediction harness

Source:
- item_id: paper_2026_xxx
- conversation_id: chat_20260426_xxx

Insight:
This paper's memory consolidation mechanism may inspire a weekly harness update step for future prediction agents. The useful part is not the storage backend, but the offline pruning and contradiction resolution loop.

Potential use:
- Compare with temporal internal feedback
- Use as related work for agent memory / harness evolution
- Consider adding “dream-style harness consolidation” as future work
同时生成一个用户画像更新候选：

candidate_profile_update:
  topic: agent memory consolidation
  reason: user asked multiple deep questions about this item
  evidence:
    - conversation_id: chat_20260426_xxx
  confidence: medium
  action: increase_topic_weight
可以让用户确认，也可以每周自动整理。

9. LLM 交互的 RAG 流程
用户问一个问题时，不要直接把问题丢给 API。应该这样：

User Question
  ↓
Intent Router
  ↓
Context Builder
  ├── 当前 item / digest
  ├── 相关论文 / repo / HN comments / X posts
  ├── 用户画像
  ├── 历史收藏
  └── 相关 research notes
  ↓
LLM Answer
  ↓
Evidence Check
  ↓
Conversation Distillation
  ↓
Memory Update Candidate
其中 Intent Router 可以识别：

1. explain
   解释这条内容

2. compare
   和其他工作对比

3. related_work
   找相关论文

4. implementation
   给复现或实现建议

5. idea_generation
   基于它生成 project idea

6. skepticism
   检查 claim 是否可靠

7. summarize_comments
   总结 HN/X 社区讨论
10. 针对 X 和 HN 的特殊处理
10.1 X：作为弱证据、早期信号
X 内容建议打上：

source_reliability: weak_to_medium
evidence_role: early_signal
needs_verification: true
X post 进入日报时，要写清楚：

这是社区早期信号，不是经过验证的论文结论。
推荐卡片可以这样：

### X Signal: 某研究者讨论了 agent memory 的新方向

**为什么值得注意**：
作者是相关论文的一作，并附带了 GitHub repo。

**可信度**：
中等。需要查看论文和 repo。

**建议动作**：
先收藏，等 arXiv/repo 完整后再深读。
10.2 HN：作为社区讨论和工程可用性信号
HN 评论适合总结：

1. 开发者认为哪里有用
2. 大家质疑什么
3. 有没有人指出实现问题
4. 是否有人给出替代方案
5. 是否出现高质量外链
HN 卡片可以这样：

### HN Discussion: 一个新的 AI coding agent 项目引发讨论

**讨论焦点**：
- 工程师关心它是否真的能处理大型 repo
- 有人质疑 demo 过于 cherry-pick
- 有人提到类似项目 A/B/C

**对你的价值**：
这可以帮助你判断该 agent 架构是否值得作为 baseline。
11. 作业创新点可以这样写
你可以把创新点升级成三个：

创新点 1：多源异质信息融合
系统不是只抓论文，而是融合：

论文：arXiv / OpenReview / Semantic Scholar
代码：GitHub
社区：HN / X
官方：公司 blog / model release
趋势：Hugging Face Papers
每类信息承担不同角色：

论文 = 正式技术贡献
GitHub = 可复现性和工程活跃度
HN = 工程社区讨论
X = 早期信号和作者解读
Blog = 官方解释和产品动态
创新点 2：交互式研究理解
日报不是静态输出，而是每条内容都可以继续问：

这是什么？
为什么重要？
和我有什么关系？
我能怎么用？
有哪些风险？
能不能形成研究选题？
这个很适合展示 Agent，而不是简单 summarizer。

创新点 3：Conversation-to-Knowledge 沉淀机制
用户和大模型的交互不会消失，而是经过蒸馏后沉淀成：

research note
topic preference
author watchlist
negative preference
future reading task
project idea
这比普通聊天机器人更强，因为它会让系统越用越懂你。

微信公众号 / 中文社区 能爬取得到吗？
我也希望每天都浏览一下当天arxiv上ai相关的所有新论文的标题摘要，不一定完全是大模型和agent

我会把这个问题拆成两块：中文信息源的“可获得性/合规性/稳定性”，以及 arXiv 每日全量 AI 论文的采集与分层阅读方案。重点不是“能不能硬爬”，而是哪些源适合作为稳定作业系统。

关于 arXiv 这一点，我建议系统分成“全量扫读”和“个性化精读”两条线：前者保证你每天看见 AI 相关新论文标题摘要，后者才按你的 LLM/Agent 偏好精选。这样不会把系统变成过窄的研究订阅器。

Thought for 1m 2s
可以，但要分层做。我的建议是：

微信公众号和中文社区可以接入，但不要把它们当作最稳定的一层；arXiv 每日 AI 全量标题摘要应该作为系统的核心稳定输入源。

1. 微信公众号能不能爬？
能获取一部分，但不建议做“无边界爬公众号”。

微信公众号有几种获取方式，稳定性和合规性差别很大。

方式	能力	稳定性	建议
自己公众号的官方素材接口	获取自己公众号后台素材	高	适合自有公众号，不适合订阅别人的公众号
WeWe RSS / we-mp-rss 这类工具	把订阅号转 RSS	中	可以做个人自用 watchlist
RSSHub / 新榜等路由	部分公众号可抓	低到中	可作为补充源
直接爬微信文章页	技术上可能，但容易反爬/失效	低	不建议作为作业系统核心
手动收藏/转发链接进入系统	最稳	高	建议保留为兜底入口
官方公众号素材接口本质上需要公众号自己的 access token，并通过 batchget_material 拉取永久素材列表，接口参数包括素材类型、offset、count，且单次 count 通常是 1 到 20；这更适合“同步自己管理的公众号文章”，不是订阅全网公众号文章。

社区方案方面，WeWe RSS 这类项目支持私有化部署并生成微信公众号 RSS，原理通常依赖微信读书/订阅链路，使用上比直接爬公众号页面稳定一些。
 但 RSSHub 的微信公众号相关路由确实存在不稳定情况，例如 2024 年就有“微信公众号无法抓取”的 open issue。

所以我建议你的系统这样设计：

微信公众号 Connector 不做成核心爬虫
而是做成 Watchlist + RSS + 手动导入 的混合入口
也就是：

1. 用户维护一个公众号白名单
2. 系统优先通过 WeWe RSS / we-mp-rss / RSSHub 获取
3. 抓不到时只记录“订阅源异常”，不阻塞日报
4. 用户可以手动粘贴公众号文章链接
5. 系统抽取标题、摘要、正文要点，进入知识库
这样作业里既能展示中文信息源能力，又不会被微信反爬拖垮。

4. arXiv 采集方式
arXiv 官方 API 支持程序化访问 arXiv 上的大量 e-print 元数据。
 它支持 search_query、start、max_results 分页，也支持按 submittedDate、lastUpdatedDate 排序。

你每天可以构造这样的 query：

(
  cat:cs.AI OR
  cat:cs.LG OR
  cat:stat.ML OR
  cat:cs.CL OR
  cat:cs.CV OR
  cat:cs.RO OR
  cat:cs.MA OR
  cat:cs.IR OR
  cat:cs.NE OR
  cat:cs.MM OR
  cat:cs.SD OR
  cat:cs.SE OR
  cat:cs.HC OR
  cat:cs.CY OR
  cat:cs.CR
)
AND submittedDate:[YYYYMMDD0000 TO YYYYMMDD2359]
arXiv API 的字段前缀包括 ti 标题、au 作者、abs 摘要、cat subject category，并支持 submittedDate:[YYYYMMDDTTTT TO YYYYMMDDTTTT] 这种 GMT 日期范围过滤。

由于 arXiv 的 submittedDate 时间范围是 GMT，你的系统应该明确处理时区：你在台北时间每天早上跑任务时，建议抓取 前一个 UTC 日 或者设置一个更宽的 36 小时窗口，然后按 arXiv ID 去重。
